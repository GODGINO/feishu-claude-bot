# 卡片交互按钮

## 语法

在回复中使用 `<<BUTTON:显示文案|操作标识|样式?>>` 添加按钮。

- **显示文案**：按钮上显示的文字
- **操作标识**：用户点击后你会收到 `[用户名 点击了按钮: 显示文案]` 的通知
- **样式**（可选）：`primary`（蓝色）、`danger`（红色），不填为默认灰色

按钮会渲染在卡片底部，用户点击后触发对应操作。

## 适用场景

### 1. 操作确认

完成代码修改、文件生成等工作后，提供后续操作按钮：

```
代码修改完成，所有测试通过。

<<BUTTON:推送代码|push|primary>>
<<BUTTON:查看 diff|show_diff>>
<<BUTTON:撤销修改|revert|danger>>
```

### 2. 链接按钮

当操作标识是一个 URL（http:// 或 https://）时，按钮会**直接在浏览器中打开链接**，不触发回调，不通知你。按钮文案会自动加上 🔗 前缀。

```
页面已部署完成！

<<BUTTON:查看页面|https://sigma.tixool.com/tunnel/xxx/index.html|primary>>
<<BUTTON:打开文档|https://docs.example.com/xxx|primary>>
```

适用于：tunnel 页面链接、外部文档、预览 URL 等用户需要直接打开的链接。

### 3. 多选项

需要用户在多个方案中选择时：

```
有两种实现方案：
- **方案 A**：使用 REST API，实现简单但性能一般
- **方案 B**：使用 WebSocket，实时性好但复杂度高

<<BUTTON:选择方案 A|choose_plan_a|primary>>
<<BUTTON:选择方案 B|choose_plan_b|primary>>
```

### 4. 定时任务/报告后续

定时任务执行完成后，提供后续操作：

```
晨报已生成并发送。

<<BUTTON:查看完整报告|view_report|primary>>
<<BUTTON:转发给团队|forward_report>>
```

## 规则

- 按钮放在回复内容的**末尾**
- 每个回复最多 3-4 个按钮，避免过多
- 按钮文案简短（2-6 个字）
- 最重要的操作用 `primary` 样式
- 危险操作（删除、撤销）用 `danger` 样式
- 按钮是可选的增强，不是必须的——简单回复不需要按钮

## 严禁的按钮

**每个按钮都必须触发一个具体的下一步动作。** 点击后你会收到通知并执行操作，所以按钮本身必须是明确的指令。

❌ 禁止的无效按钮（点击后无法产生有意义的动作）：
- "OK" / "好的" / "收到" / "确认" — 纯确认无动作
- "可以用" / "没问题" / "满意" — 表达态度，不是指令
- "还要改" / "有修改意见" / "不满意" — 模糊，你不知道要改什么
- "继续" / "下一步" — 没有指定继续做什么
- "了解" / "明白了" — 无后续操作

✅ 正确的按钮（每个都是可直接执行的具体指令）：
- "提高分辨率到600DPI" — 明确操作
- "推送代码" / "部署到生产" — 具体动作
- "生成移动端版本" — 新的明确任务
- "发送给@Amanda" — 指定接收人
- "转为PDF格式" — 格式转换指令

如果没有有价值的后续操作，就不要加按钮。宁可不加，也不要凑数。
